# Журнал решений — проект "Выбиратор"

## 2026-07-16 — Архитектура барабана

**Решение:** Ядро барабана не хранит список патронов в коде — читает отдельный конфигурационный справочник. Новый патрон = новая строка в справочнике, без правки движка.

**Обоснование:** Config-driven architecture: рост числа сценариев не должен требовать переписывания ядра.

## 2026-07-16 — Нормализация шкал

**Решение:** Любой критерий внутри патрона обязательно приводится к единой внутренней шкале (0-100) перед расчётом коэффициента.

**Обоснование:** Разные шкалы (рубли, баллы, годы) нельзя складывать напрямую без нормализации — иначе результат искажается одним доминирующим критерием.

## 2026-07-16 — Совмещение сценариев

**Решение:** Сценарии разных категорий не складываются механически в одну сумму баллов. Совмещение (если понадобится) — отдельный, специально спроектированный сценарий с собственной целью.

**Обоснование:** Риск конфликтующих ранжирований и фальшивой точности при слиянии разнородных critериев.

## 2026-07-16 — Интегрированный коэффициент

**Решение:** Для сравнения между категориями нужен безразмерный нормированный коэффициент по аналогии с финансовыми коэффициентами рентабельности (не зависит от масштаба).

**Обоснование:** Абсолютные величины (рубли, часы) не сравнимы между категориями напрямую.

## 2026-07-16 — Тумблер контекста

**Решение:** Тумблер "для себя / для бизнеса" — уровень "закона" над патронами. Меняет макро-параметры и формулу (например, налоги для юрлиц), а не переписывает патрон.

**Обоснование:** Иерархия: ядро (конституция) -> контекст (закон) -> патрон (приказ) -> карточки товаров (инструкция).

## 2026-07-16 — Геометрия диска набора номера

**Решение:** 11 секторов (10 цифр + 1 пустой), ограничитель зафиксирован у цифры 0 (позиция "на 4 часа"), пустой сектор между "1" и ограничителем. Вращение по часовой при наборе, обратно при возврате.

**Обоснование:** Соответствие реальной механике дискового телефона по образцу пользователя.

## 2026-07-16 — Укрупнённая структура категорий

**Решение:** 1-недвижимость, 2-движимость, 3-бытовая техника/электроника (пылесос как готовый патрон), 4-люди/услуги, 5-товары для людей, 6-животные, 7-9 резерв, 0-настройки.

**Обоснование:** Электроника/техника сравнивается чаще всего, авто и недвижимость — самые дорогие решения. Файл category_tree.xlsx (28 категорий, 8472 позиции) избыточно детален для верхнего уровня.

## 2026-07-16 — Хранение данных

**Решение:** Черновое редактирование данных — в Google Таблицах (папка "Выбиратор"). Финальные CSV для кода — в репозитории GitHub (vybirator/data/), т.к. Google Диск блокируется CORS при прямом обращении из браузера.

**Обоснование:** GitHub раздаёт статичные файлы с правильными CORS-заголовками, Google Диск — нет.

## 2026-07-16 — Эталон данных по пылесосам

**Решение:** Файл vacuum_items.csv (11 моделей, с расчётной стоимостью уборки) зафиксирован как эталонная версия данных патрона "пылесосы" на эту дату.

**Обоснование:** Патрон дорабатывался несколько раз в процессе, нужна явная точка отсчёта перед переносом в постоянное хранилище.
