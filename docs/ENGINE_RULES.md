Документ описывает единый стандарт структуры и логики для всех патронов
(vacuum, phones, и всех последующих). Создан 20.07.2026.

## 1. Структура репозитория

```
vybirator/
├── data/                        # ВСЕ CSV-файлы с данными моделей — ОБЩАЯ папка для всех патронов
│   ├── vacuum_items.csv
│   ├── phone_items.csv
│   └── <name>_items.csv         # для будущих патронов
├── patrons/                     # ВСЕ HTML-страницы (движки) — каждый патрон в своей подпапке
│   ├── vacuum/
│   │   └── vacuum.html
│   └── phones/
│       └── phones.html
└── docs/
    └── ENGINE_RULES.md          # этот файл
```

Правило: **данные (CSV) — в `data/`, движок (HTML) — в `patrons/<name>/`.**
Никогда не хранить CSV внутри `patrons/<name>/` и не "зашивать" данные
прямо в JavaScript внутри HTML — это ломает единообразие и историю Git.

## 2. Именование файлов

- CSV с моделями: `<name>_items.csv` (например `vacuum_items.csv`, `phone_items.csv`).
- Доп. файлы весов/пресетов: `<name>_<назначение>.csv`

  чтобы избежать путаницы при поиске файлов по маске).
- HTML-страница патрона: `<name>.html`, лежит в `patrons/<name>/<name>.html`.

## 3. Как HTML должен загружать данные

Внутри `<name>.html` обязательно используется относительный fetch,
поднимающийся на уровень выше в `data/`:

```javascript
fetch('../data/<name>_items.csv')
  .then(r => r.text())
  .then(csvText => { /* парсинг CSV и рендер */ })
  .catch(err => {
    document.getElementById('controls').innerHTML =
      `<div class="error">Не удалось загрузить данные (${err.message}).
       Проверь, что файл лежит в data/ и сайт открыт через GitHub Pages,
       а не как локальный файл.</div>`;
  });
```

Важно: fetch с относительным путём работает только через HTTP(S)
(например GitHub Pages), а не при открытии файла напрямую с диска
(`file://`) — при локальном открытии браузер блокирует fetch на CSV
из соображений безопасности (CORS). Для локальной проверки нужен
локальный веб-сервер (например `python -m http.server`).

## 4. Общие элементы движка (единые для всех патронов)

- Ползунки (sliders) критериев — от 0 до 10, с флагом инверсии
  (чем меньше значение — тем выше балл, например компактность).
- Пьедестал (podium) — топ-3 модели по сумме взвешенных баллов,
  пересчитывается в реальном времени при движении любого ползунка.
- Фильтры (dropdown/checkbox) — жёсткий отбор по категориальным
  признакам (бюджет, формфактор, бренд и т.п.), применяется ДО расчёта
  баллов ползунков.
- Пресеты весов — кнопки, задающие стартовые значения ползунков под
  конкретный сценарий использования (например возрастная группа).
- Подвал (footer) — текстовый блок с допущениями и ограничениями
  данных, обязателен для каждого патрона.

## 5. Допущения и ограничения (общие для всех патронов)

3. Категориальные фильтры могут быть неполными на старте (не все
   значения представлены моделями) — это нормально для первой

5. Адаптировать список критериев/ползунков и подвал под новый патрон.
6. git add + commit + push всех новых файлов.
7. Обновить этот документ, если появилось новое общее правило.
